# 📐 턴 데이터 추출 기준 (Extraction Criteria)

> `src/extract_turns.py` + `src/scoreboard.py` 가 영상에서 턴 데이터를 뽑는 **판정 기준 명세**.
> **대원칙: 턴 자체와 라벨(수구·성공·뱅크샷·턴 교대)을 전부 방송 점수판이 결정한다.**
> 영상(비전)은 "공이 어디에 있고 언제 멈췄는가"(좌표·타이밍)만 담당한다.

---

## 0. 점수판 우선 원칙 — 항목별 근거 소스

| 추출 항목 | 근거 | 방식 |
|---|---|---|
| **턴 정의(분할)** | 🟡 점수판 | **이닝 원형 이벤트**의 연속한 두 이벤트 = 한 턴. 영상 상태머신이 아니라 점수판이 턴 경계를 만든다 |
| **수구(shooter)** | 🟡 점수판 | 이닝 원형에 **숫자가 떠 있는 색** = 지금 치는 선수 |
| **턴 교대** | 🟡 점수판 | 숫자가 **상대 색 원으로 이동**(상대 원에 0부터) → 색 전환 = 이전 선수 실패 |
| **성공/실패** | 🟡 점수판 | 수구 색의 **이닝 원형 +1↑** 또는 **총점 박스 +1~2** (두 지표 교차) |
| **뱅크샷** | 🟡 점수판 | 이닝 원형 **+2**(직접 관측) 또는 이닝 판독이 없을 때 총점 박스 **+2** |
| 공 위치(before/after) | 🎥 영상 | 점수판으로는 알 수 없음 — 탑뷰 YOLO 검출. **정지 배치 → 없으면 공별 최근접 관측** |
| 좌표 품질 | 🎥 영상 | `before_source`(still/obs_near/obs_far) + `coverage`로 표시 |

> **핵심 변경(2026-07):** 예전엔 영상이 `정지→샷→정지`를 완성해야 턴이 생겼고 점수판은 채점만 했다.
> 지금은 **점수판 이닝 이벤트가 턴을 정의**하고, 영상은 그 턴의 좌표만 채운다.
> → 추적이 끊겨 영상이 샷을 놓쳐도 **점수판 변화가 있으면 턴이 확정**된다.
> DB 로더(`db/load_to_db.py`)는 `method="scoreboard"` 턴만 적재하므로 DB의 모든 라벨은 점수판 근거다.

## 1. 점수판 읽기 방식 (`src/scoreboard.py`)

PBA 점수판 구조 (예: 32강 화면):

```
 ┌────────────────────────────────┐
 │  김영원   [ 6 ] (⚪  )  ≡       │   ← 총점 박스 6, 원형: 숫자 없음 = 대기 (수구 아님)
 │  초클루   [ 4 ] (🟡 3)  ≡       │   ← 총점 박스 4, 원형: 3 = 현재 이닝 3득점 중 (수구=노랑)
 └────────────────────────────────┘
```

- **총점 박스**: 각 선수 누적 점수 (박스 색 = 그 선수의 수구 색). 큰 숫자라 판독 신뢰도 높음.
- **이닝 원형**: 지금 치는 선수의 원에만 숫자(이번 이닝 연속 득점)가 뜬다.
  **숫자가 떠 있는 원의 색 = 수구.** 실패로 턴이 넘어가면 숫자가 상대 원으로 이동(0부터).
- 판독기 산출물:
  - `events`: `[(frame, 흰총점, 노란총점)]` — 총점이 바뀐 시점
  - `active_events`: `[(frame, 수구색, 이닝점수)]` — 이닝 원형이 바뀐 시점 = **턴 경계**
- 판독 신뢰성 장치:
  - **2회 연속 동일 판독**일 때만 확정 (한 프레임 오독 무시)
  - 두 원이 **둘 다 숫자거나 둘 다 없으면 "모호"** → 보류
  - 총점 40 초과 판독은 오독으로 폐기 (`SCORE_MAX`)
  - 샘플 주기: 처리 5프레임마다 + **공이 멈춘 순간마다 즉시** (§3 참고), **탑뷰가 아니어도** 읽는다

## 2. 프레임 사용/폐기 기준

```
모든 프레임 ─┬─ 점수판 OCR (5프레임마다 + 공 정지 순간, 화면 종류 무관)
             │
             ├─ [관문1] 탑뷰 형태? (가로 2:1 · 수평 변 · 화면 12~80% · 경계 미접촉)
             ├─ [관문2] 기준 꼭짓점(초기 탑뷰 8프레임 중앙값 합의)과 90px 이내?
             │     실패 → 클로즈업/컷 → 공 검출 없이 버림
             └─ [관문3] YOLO 공 검출 — conf 0.45 미만 버림, 색깔별 최고 1개만
```

| 폐기 규칙 | 기준 |
|---|---|
| 카메라 컷/클로즈업 | 관문1·2 실패 프레임 (공 검출 안 함) |
| 편집 점프 컷 | 공이 프레임당 0.3m(9m/s) 이상 "순간이동" → 진행 중 정지 구간 끊고 리셋 |
| 클립 경계 | 탑뷰 4초 이상 끊김 → 리셋, `epoch` 증가 |
| 저신뢰 탐지 | YOLO conf < 0.45 |

> 이 관문들은 **좌표를 채울 수 있는지**를 가른다. 관문에서 버려진 프레임이 많아도(추적 성김)
> 턴 자체는 점수판이 만들므로 사라지지 않는다 — 좌표만 근사치가 되고 `coverage`가 낮게 찍힌다.

## 3. 좌표 수집 — 영상 담당 (턴을 만들지는 않음)

영상 패스는 턴을 **정의하지 않고**, 점수판 턴이 좌표를 가져다 쓸 **정지 배치 타임라인**을 쌓는다.

- **정지 배치**: 세 공 각각 최근 12개 관측 중 8개 이상이 0.008(≈2cm) 범위 + 1.5초 내 관측. 좌표는 중앙값.
  같은 배치가 이어지면 한 구간으로 묶어 `still_log = [(f_start, f_last, layout)]` 에 기록.
- **공별 관측**: 모든 탑뷰 프레임의 공별 좌표를 `obs` 에 누적 (정지 배치가 없을 때의 폴백용).
- **정지 순간 동기 스냅샷** ★: 공이 멈춰 있는 프레임에서는 **그 즉시 점수판을 읽는다**
  (`extractor._cur_still` 이 이 프레임까지 이어질 때 `score_reader.sample()` 호출).
  → 좌표와 점수판(수구·이닝·총점)을 **같은 시점**에 확보 → 중간 판독 누락으로 인한 오판 방지.
  (`sample()` 내부의 픽셀 변화 게이트 덕분에 실제 OCR은 값이 바뀔 때만 돈다 — 비용 거의 없음.)

## 4. 턴 분할 + 성공/뱅크샷 판정 — 점수판 담당

`build_turns_from_scoreboard()` 가 **이닝 원형 이벤트(`active_events`)로 턴을 만든다.**

### 4-1. 턴 경계
연속한 두 이벤트 `ev[i] → ev[i+1]` = 한 턴.
- `shooter` = `ev[i]` 의 색, `frame_start/end` = 두 이벤트의 프레임
- `before` = `ev[i].frame` 시점 좌표, `after` = `ev[i+1].frame` 시점 좌표
  (§3의 정지 배치 우선 → 없으면 공별 최근접 관측. 소스는 `before_source`/`after_pos_source`)

### 4-2. 성공/실패 (두 지표 교차)
| 판정 | 기준 |
|---|---|
| **성공** | 다음 이벤트가 **같은 색이며 이닝 원형 증가**(득점) **또는** 창의 **총점 박스가 +1~2** |
| **실패** | 색이 바뀜(턴 교대) 이면서 득점 근거 없음 |
| **뱅크샷** | 이닝 원형 **+2**(직접 관측) · 또는 이닝 판독이 없을 때 총점 **+2** |

- **총점 지연 보정**: 총점 박스 반영이 이닝 이벤트보다 살짝 늦다(오퍼레이터 입력 지연).
  판정 창의 양 끝을 동일하게 `SCORE_FAIL_MARGIN_S`(3초)만큼 밀어 비교한다
  (안 그러면 직전 턴 득점이 다음 턴에 잘못 귀속됨).
- 뱅크샷 규칙 근거: **이닝 원형이 +2를 직접 봤으면 확정.** 총점 +2만 있고 이닝이 +1을 명확히
  봤다면 일반 득점(총점 +2는 이전 놓친 득점의 몰아반영일 수 있어 뱅크로 치지 않음).

### 4-3. 감사(audit) 필드
판정 근거를 사후 확인할 수 있게 `success_detail` 에 다음을 남긴다:
`run_from`/`run_to`(이닝 전/후), `total_delta`(총점 증가), `totals`(전/후 총점 `[흰,노랑]`),
`before_source`/`after_pos_source`, `coverage`.

### 4-4. 폴백 / 폐기
- **이닝 이벤트가 2개 미만**(원형 판독 실패)인 방송 → 구(舊) 방식 폴백:
  영상 상태머신 턴 + 총점 박스 판정(`apply_scoreboard_judgment`) 후 리플레이·판정불가 턴 폐기.
- **점수판이 아예 없는 영상**(스포방지 마스킹·화면기록 등) → 라벨 신뢰 불가, **전 턴 폐기**(0턴).

## 5. 실시간 분석에서의 대응 관계

실시간 워커(hybrid-engine: `realtime_tracker` + `minsu_detector` + `scoreboard_reader`)도 같은 원칙:

```
스트림 매 프레임 → 탑뷰 검사 ─ 실패 → 스킵 ("지원하지 않는 카메라 화면")
                     ↓ 통과
              공 움직임 추적 ─ 움직이는 중 → 분석 안 함 ("공 정지 대기")
                     ↓ 세 공 정지 ★ 이 순간에만 분석 진입 + 점수판 판독
              배치 확정 → 확률 엔진 호출 → 확률 표시
```

- **분석·점수판 확보가 실행되는 프레임 = "탑뷰 + 세 공 정지"가 성립한 순간뿐** — 배치 파이프라인의
  "정지 순간 동기 스냅샷"(§3)과 같은 발상이다.
- 차이는 판정 시점: 배치는 다음 이벤트까지 보고 **사후 확정**, 실시간은 멈춘 순간 **즉시** 확률을 낸다.
